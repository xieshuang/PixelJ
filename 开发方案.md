这份架构设计文档将从“Java专家”的视角出发，侧重于**高性能、低延迟、高扩展性**。我们要构建的不是一个简单的 Demo，而是一个能流畅处理上万张单反原片（RAW/4K JPEG）的工业级工具。

## 个人电脑照片浏览器 (PixelJ) 架构设计文档

### 1. 设计目标 (Design Goals)

- **极致响应：** 界面操作不卡顿（UI 线程与 IO 线程分离）。

- **内存可控：** 即使管理 100GB 的照片，JVM 堆内存占用也应保持在合理范围（如 512MB 内）。

- **平滑滚动：** 瀑布流预览支持 $60fps$ 帧率。

- **格式扩展：** 插件化支持 RAW、WebP、HEIC 等格式。

---

### 2. 核心架构图

系统采用分层架构，核心在于**异步任务编排**和**多级缓存策略**。

### 3. 详细模块设计

#### 3.1 异步加载引擎 (Async Loading Engine)

这是整个系统的“心脏”。我们不能在 UI 线程加载图片。

- **技术实现：** 基于 `CompletableFuture` 或 `ExecutorService` 构建自定义线程池。

- **优先级队列：** 优先加载用户当前屏幕可见的照片，次要加载预取（Pre-fetch）的照片。

- **源码细节：** 使用 `ThreadFactory` 将线程命名为 `image-decoder-pool-%d`，并设置较低的优先级 `Thread.MIN_PRIORITY`，确保不会抢占主界面的渲染资源。

#### 3.2 多级缓存系统 (Multi-level Caching)

解决“重复加载耗时”和“内存溢出”的矛盾。

| **缓存层级**    | **存储介质** | **内容**               | **淘汰算法**                |
| ----------- | -------- | -------------------- | ----------------------- |
| **L1 (内存)** | JVM Heap | `Image` 对象 (Bitmap)  | **LRU + SoftReference** |
| **L2 (磁盘)** | 本地临时目录   | 压缩后的缩略图 (Thumbnails) | 空间阈值清理                  |
| **L3 (原始)** | 用户磁盘     | 原始照片文件               | 无                       |

> **专家提示：** 缓存 `Image` 对象时，务必记录其占用的内存大小。一张 $4000 \times 3000$ 的图片，解码后占用的内存约为 $4000 \times 3000 \times 4$ 字节 $\approx 48MB$。

#### 3.3 视图层设计 (View Layer)

- **JavaFX 虚拟化：** 使用 `FlowPane` 配合 `ScrollPane` 时，必须实现**可视区域检测**。只有出现在 Viewport 内的图片才触发 `L1` 缓存加载，离开区域后立即释放 `Image` 引用。

- **硬件加速：** 开启 `-Dprism.order=d3d,sw`（Windows）以利用显卡进行纹理渲染。

---

### 4. 核心流程：打开文件夹

当用户选择一个包含 5000 张照片的文件夹时，后台发生的动作：

1. **扫描 (Scanning)：** `NIO` 递归扫描文件树，获取 `Path` 列表。

2. **监听 (Watching)：** 启动 `WatchService` 监控该目录，实时捕获文件增删。

3. **索引 (Indexing)：** 将基本元数据（文件名、修改时间、Exif 信息）存入轻量级内存数据库（如 **SQLite** 或 **H2**），方便快速排序和搜索。

4. **分批渲染 (Batch Rendering)：** UI 仅加载前 50 个占位符，由后台线程池滚动更新缩略图。

### 6. 扩展性考虑 (SPI 机制)

为了支持更多图片格式（比如单反的 `.CR2` 或 `.NEF`），我们可以利用 Java 的 **SPI (Service Provider Interface)**。

- 定义 `ImageDecoder` 接口。

- 不同的解码实现（如基于 `ImageIO` 或 `Apache Commons Imaging`）通过 `META-INF/services` 注入，实现真正的零耦合插件化。
